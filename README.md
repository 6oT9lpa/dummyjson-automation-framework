# DummyJSON API Automation Framework

Проект представляет собой фреймворк для автоматизированного API-тестирования публичного сервиса [DummyJSON](https://dummyjson.com).

В рамках задания реализована структура API automation framework: конфигурация вынесена в отдельный файл, тестовые данные валидируются через Pydantic-схемы, HTTP-запросы инкапсулированы в API-клиентах, а проверки написаны на `pytest`.

## Опыт тестирования

### Ручное тестирование

Есть базовый опыт ручного тестирования:

- анализ требований;
- составление чек-листов и тест-кейсов;
- выполнение позитивных и негативных проверок;
- заведение дефектов;
- проверка API через ручные инструменты, например Postman или аналогичные клиенты.

### Нагрузочное тестирование

Есть базовое понимание нагрузочного тестирования:

- проверка времени ответа;
- анализ стабильности системы под нагрузкой;
- контроль количества успешных и ошибочных запросов;
- понимание метрик throughput, response time, error rate.

В рамках данного задания нагрузочные тесты не реализовывались, так как задача была сфокусирована на функциональном API-тестировании.

### Тестирование безопасности

Есть базовое понимание тестирования безопасности API:

- проверка авторизации;
- проверка работы access token;
- негативные проверки без токена;
- проверка реакции API на неверные credentials;
- базовый контроль доступа к защищённым endpoints.

В рамках задания реализованы базовые security-related проверки, связанные с авторизацией и доступом к `/auth/me`.

## Конфигурация

Основные настройки проекта находятся в файле:

```text
data/config.json
```

Пример конфигурации:

```json
{
  "base_url": "https://dummyjson.com",
  "timeout": 10
}
```

Конфигурация загружается через `ConfigManager`, после чего используется в базовом API-клиенте.

## Тестовые данные

Тестовые данные находятся в файле:

```text
data/test_data.json
```

В файле хранятся данные для:

- авторизации;
- негативной проверки авторизации;
- проверки текущего пользователя;
- работы с корзинами;
- примеров пользовательских тестовых данных.

Тестовые данные валидируются через Pydantic-модели. Это позволяет заранее проверять корректность структуры данных и типов до выполнения тестов.

Пример пользовательских тестовых данных:

```json
{
  "first_name": "Alex",
  "last_name": "Tester",
  "email": "alex.tester@example.com",
  "age": "30",
  "salary": "5000",
  "department": "Quality Assurance"
}
```

## Архитектура

### BaseApiClient

`BaseApiClient` отвечает за общую HTTP-логику:

- построение URL;
- выполнение `GET`, `POST`, `PUT`, `PATCH`, `DELETE` запросов;
- работу с `requests.Session`;
- использование таймаута из конфигурации;
- логирование запросов и ответов;
- формирование Bearer-заголовков для авторизованных запросов.

### AuthApiClient

`AuthApiClient` содержит методы для работы с авторизацией:

- `POST /auth/login`
- `GET /auth/me`

### CartsApiClient

`CartsApiClient` содержит методы для работы с корзинами:

- `GET /carts/user/{userId}`
- `GET /carts/{cartId}`
- `POST /carts/add`
- `PUT /carts/{cartId}`
- `DELETE /carts/{cartId}`

### ApiResponse

`ApiResponse` является обёрткой над HTTP-ответом и предоставляет:

- доступ к status code;
- получение JSON-тела ответа;
- преобразование ответа в Pydantic-модель;
- проверку ожидаемого HTTP-статуса.

## Реализованные проверки

### Auth API

- `POST /auth/login` — успешная авторизация пользователя с валидными данными.
- `POST /auth/login` — негативная проверка авторизации с неверным паролем.
- `GET /auth/me` — получение данных текущего пользователя с Bearer token.
- `GET /auth/me` — негативная проверка запроса без токена.

### Carts API

- `GET /carts/user/{userId}` — получение списка корзин конкретного пользователя.
- `GET /carts/{cartId}` — получение корзины по ID.
- `POST /carts/add` — создание новой корзины.
- `PUT /carts/{cartId}` — обновление корзины.
- `DELETE /carts/{cartId}` — удаление корзины.
- `GET /carts/{cartId}` — негативная проверка запроса несуществующей корзины.

## Особенности DummyJSON

DummyJSON является тестовым API и имитирует изменение данных.

Для запросов `POST`, `PUT` и `DELETE` сервис возвращает ответ, похожий на результат изменения данных, но фактически не сохраняет изменения на сервере. Поэтому проверки создания, обновления и удаления корзины валидируют контракт ответа конкретного запроса, а не последующее состояние данных в системе.

## Что проверяется в тестах

В тестах проверяются:

- HTTP-статусы ответов;
- структура JSON-ответов;
- соответствие ответа Pydantic-схемам;
- наличие обязательных полей;
- корректность авторизации;
- обработка негативных сценариев;
- работа API-клиентов с тестовыми данными;
- корректная сериализация request body через Pydantic-модели.

## Итог

Проект демонстрирует базовый подход к построению API automation framework на Python:

- код разделён на слои;
- конфигурация вынесена отдельно;
- тестовые данные отделены от тестовой логики;
- request и response контракты описаны через Pydantic;
- API-запросы инкапсулированы в клиентах;
- тесты можно запускать стандартной командой `pytest`.
